# Fundamentals

## Introduction

Accessibility is a foundational aspect of digital product design.

It ensures that people with a wide range of abilities — including those with visual, auditory, motor, and cognitive disabilities — can perceive, understand, navigate, and interact with digital interfaces effectively.

Accessibility is not an additional feature or a later enhancement. It is a core quality of a well-designed product.

## The bigger picture

Accessibility issues are often invisible during design, but highly visible in real use.

When a button has no accessible label, a form lacks clear validation, or an interface cannot be navigated using a keyboard, users may be unable to complete essential tasks.

This is not a matter of convenience — it directly impacts access.

- 1.3 billion people worldwide live with a disability (WHO)
- 97% of websites fail basic accessibility checks (WebAIM)
- 71% of users with disabilities leave inaccessible websites

Accessibility is about identifying and removing these barriers across the entire experience.

## Why accessibility matters

### It supports inclusion

Digital products are essential for communication, services, and participation in society. Accessibility ensures that these products can be used by the widest possible audience.

### It is often a legal requirement

In many regions, accessibility is enforced through regulations:

- **ADA (United States)** requires equal access to services, including digital experiences
- **EN 301 549 (European Union)** defines accessibility requirements for ICT products and services

These frameworks increasingly apply to websites, applications, and internal tools.

### It improves overall usability

Practices such as clear structure, readable content, and predictable interaction benefit all users — including those in temporary or situational constraints (e.g. low light, distractions, limited mobility).

## Core principles (POUR)

Accessibility guidelines are structured around four principles defined by WCAG.

### 1. Perceivable

Information must be presented in ways users can perceive.

This includes:

- Text alternatives for images and icons
- Sufficient contrast between text and background
- Captions or transcripts for media
- Avoiding reliance on color alone to convey meaning

If content cannot be perceived, it is effectively inaccessible.

### 2. Operable

Users must be able to interact with all interface elements.

This includes:

- Full keyboard navigation
- Visible and consistent focus indicators
- No keyboard traps
- Sufficient time to complete tasks

Interfaces that require specific input methods (e.g. mouse only) exclude users.

### 3. Understandable

Content and interactions must be clear and predictable.

This includes:

- Clear labels and instructions
- Consistent navigation patterns
- Informative error messages with guidance
- Avoiding unexpected behavior changes

Users should not need to guess how something works.

### 4. Robust

Content must be compatible with different devices, browsers, and assistive technologies.

This includes:

- Proper semantic HTML structure
- Correct use of roles and attributes
- Ensuring compatibility with screen readers and other tools

A robust implementation ensures long-term accessibility.

## Accessibility standards and frameworks

### WCAG (Web Content Accessibility Guidelines)

WCAG is the primary international standard for digital accessibility, developed by the W3C.

It provides:

- A structured framework based on the POUR principles
- Testable success criteria
- Clear levels of conformance (A, AA, AAA)

WCAG does not prescribe specific design solutions. Instead, it defines what must be true for content to be accessible.

For most teams, WCAG is the main reference for both design and development decisions.

### Conformance levels

WCAG defines three levels of accessibility:

- **Level A** — Addresses the most critical barriers
- **Level AA** — Covers the most common and impactful issues (recommended standard)
- **Level AAA** — Highest level, not always feasible across all content

Most organizations aim to meet WCAG 2.2 Level AA, as it balances impact and practicality.

### EN 301 549

EN 301 549 is the European standard for accessibility of ICT (Information and Communication Technology).

It:

- Extends accessibility requirements beyond websites to apps, software, and documents
- Aligns closely with WCAG for web content
- Is commonly used in public sector procurement and compliance

It is particularly relevant for organizations operating in Europe or working with government-related services.

### ADA (Americans with Disabilities Act)

The ADA is a civil rights law in the United States that prohibits discrimination based on disability.

While originally focused on physical spaces, it has been interpreted to apply to digital services, including websites and applications.

In practice:

- Organizations must ensure that their digital experiences are accessible
- WCAG is commonly used as the technical reference for compliance

### ARIA (Accessible Rich Internet Applications)

ARIA is a set of attributes used to enhance accessibility in dynamic or custom interfaces.

It is used to:

- Provide additional semantic information to assistive technologies
- Define roles, states, and properties of UI components

However:

- ARIA should not replace native HTML
- Incorrect use of ARIA can reduce accessibility rather than improve it

**Best practice:** use native HTML whenever possible, and ARIA only when necessary.

## Accessibility as a shared responsibility

Accessibility spans across the entire product lifecycle.

### Design

Defines structure, hierarchy, interaction patterns, and visual clarity.

### Content

Ensures clarity, meaning, and guidance through labels, instructions, and messaging.

### Development

Implements semantics, keyboard behavior, focus management, and assistive compatibility.

### QA

Validates behavior, interaction, and compliance through manual and automated testing.

Accessibility cannot be ensured by a single role — it requires coordination.

## Common misconceptions

- Accessibility is only about visual impairments
- Accessibility is limited to color contrast
- Automated tools are sufficient for validation
- ARIA can replace semantic HTML
- Accessibility can be addressed at the end of a project

These assumptions often lead to incomplete or ineffective solutions.

## Practical baseline for teams

A consistent approach to accessibility typically includes:

- Using WCAG 2.2 AA as a baseline
- Ensuring full keyboard operability
- Providing visible focus states
- Using semantic HTML structures
- Writing clear and meaningful labels
- Avoiding reliance on color alone
- Including accessibility checks in design and QA processes

## How to validate

Accessibility should be validated using a combination of:

- Automated tools (contrast, structure, linting)
- Manual testing (keyboard navigation, interaction flows)
- Assistive technologies when possible
